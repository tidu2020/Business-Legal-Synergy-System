## 2.4 模块依赖关系

```
auth/routes.py ──→ app.py ──→ 前端
business/routes.py ──→ ai/orchestrator ──→ knowledge/models.py
legal/routes.py ──→ knowledge/models.py
                 └─→ legal/validator.py ──→ ai/orchestrator
ai/orchestrator.py ──→ ai/analyzer.py + ai/llm_client.py + knowledge/models.py
knowledge/routes.py ──→ knowledge/models.py
report/routes.py ──→ work_order_store + knowledge_base
archive/routes.py ──→ work_order_store

file_review/routes.py ──→ file_review/config.py（自带 AppConfig）
                      └─→ file_review/src/pipeline.py（自带 LLMClient）
                      └─→ backend/auth/decorator.py（仅鉴权依赖）
                      └─→ backend/law_search/（LegalSearcher 调用）

law_search/routes.py ──→ backend/law_search/flk_client.py（FLK API）
                     └─→ backend/law_search/verifier.py（法条验证）

✱ file_review 和 law_search 模块均完全独立，不依赖主系统 ai/knowledge/business/legal 模块；
  app.py 仅在启动时注册蓝图，不注入外部依赖。
```

**依赖注入**：所有 store/kb/orchestrator 在 `app.py` 的 `create_app()` 中初始化，通过各路由模块的 `init_dependencies()` 注入，实现高内聚低耦合。**file_review 模块例外**：其依赖（LLMClient/AppConfig）在模块内部创建，仅在 `app.py` 中创建 output 目录并注册蓝图，不接收外部注入。

## 2.5 数据结构设计

### 2.5.1 FAQ 知识条目（核心数据结构）

```json
{
  "id": "FAQ-202312-001",
  "case_id": null,
  "month": "202312",
  "question": "...",
  "legal_answer": "...",
  "compliance_risk": "...",
  "practical_advice": "...",
  "legal_basis": "《法名》第X条",
  "source": "FAQ整理-最终版.md",
  "source_work_order_id": null,
  "category": "知识产权-著作权",
  "tags": ["著作权", "登记"],
  "status": "confirmed",
  "created_by": "system",
  "created_at": "2023-12-01T00:00:00"
}
```

**字段设计理由**：
- `id`：唯一标识，用于检索引用
- `case_id`：案例链关联（如 Q64-67 同属"商场滑倒"案例）
- `month`：时间维度，用于归档和趋势分析
- 四段式字段：对应业务需求
- `source` / `source_work_order_id`：可追溯，建立"知识→工单→原始材料"追溯链
- `status`：draft（未确认）/ confirmed（已入库）
- `tags`：用于 KeywordMatcher 倒排索引

### 2.5.2 WorkOrder（含留痕字段）

```python
class WorkOrder:
    # 业务侧数据
    materials: list       # 上传材料
    dialogue: list        # 对话记录（业务与 AI）
    ai_conclusion: str    # AI 结论
    # 留痕字段
    id: str
    submitter: str        # 提交人
    submitted_at: str     # 提交时间
    ai_reviewed_at: str   # AI 审核时间
    # 法务侧数据
    reviewer: str          # 审核人
    reviewed_at: str      # 审核时间
    legal_conclusion: str # 法务结论
    confirmed_at: str     # 确认时间
    confirmed_faqs: list  # 入库条目
```

## 2.6 数据存储

MVP 阶段全部本地 JSON 文件存储：

```
data/
├─ accounts.json          # 账户（3 角色）
├─ knowledge_base.json   # 知识库（核心，70 条）
├─ work_orders/           # 工单（每单一文件）
│   └─ WO-YYYYMM-NNN.json
└─ archive/               # 归档包
    └─ WO-YYYYMM-NNN.md
```

**设计考量**：JSON 存储无依赖、易演示、易调试，符合课程要求；存储层已抽象（KnowledgeBase 类），未来可替换为 SQLite/向量库。

## 2.7 前端架构

三套角色界面差异化设计：

| 界面 | 文件 | 风格 |
|------|------|------|
| 登录页 | login.html | 居中表单，演示账号提示 |

公共资源：`frontend/static/` 下 `style.css`（统一配色）与 `common.js`（fetch 封装、鉴权、跳转）。
