# 01 系统需求分析

***

## 1.1 项目背景

国有企业在法务合规管理中日常面临两类痛点：

- **业务侧**：日常法务咨询（如"合同相对方要求第三方付款可以吗""证据逾期提交后果"）缺少前置自助渠道，小事也需打扰法务；正式材料（合同/对外函件/采购/上会议题）审核流转缺乏统一入口。
- **法务侧**：审核经验以文档形式散落，难以沉淀复用；年度法治工作总结、合规季报、风险报告等公文重复起草；缺乏对历史审核数据的结构化利用，难以做风险监测与年度风险识别。

本系统构建"业务-法务协同 + AI 辅助审核 + 知识自动沉淀"的前后端分离 Web 闭环平台，支撑国有企业法务合规"三个100%"（重大决策/规章制度/合同法律审核率 100%）与"四位一体"（法务/合规/内控/风控融合）体系落地。

***

## 1.3 功能需求

系统采用前后端分离架构，**每个功能模块均包含独立的后端 API 和前端页面**。业务、法务、管理员三套前端界面各不相同。

### 核心模块

| 编号 | 模块             | 后端 API               | 前端            | 说明                                         |
| -- | -------------- | -------------------- | ------------- | ------------------------------------------ |
| ①  | 账户与权限          | `/api/auth/*`        | 登录页           | 三角色登录、鉴权                                   |
| ②  | 业务门户           | `/api/business/*`    | 业务端           | 发起会话、上传材料、查看 AI 意见、提交工单、**撤回工单**           |
| ③  | 法务工作台          | `/api/legal/*`       | 法务端           | 接收工单、审核、归档前审核修改、校验、确认入库、**修改归档、提交知识库修改申请** |
| ④  | AI 审核引擎        | `/api/ai/review`     | （被业务门户调用）     | 双算法检索 + **三级 LLM 兜底** + 用户反馈、多轮对话、免责声明     |
| ⑤  | 知识库管理          | `/api/kb/*`          | 管理员端          | 管理员直接管理知识库；法务仅检索调用；**相似条目合并**              |
| ⑥  | 公文生成           | `/api/report/*`      | 法务/管理员        | 基础版：1-2 类公文                                |
| ⑦  | 归档导出           | `/api/archive/*`     | 管理员/法务        | Markdown 导出 + 按案例归档 + **退回归档**             |
| ⑧  | **合同审核**（新增）   | `/api/file_review/*` | 业务端           | 六阶段流水线 + 多轮问答，**模块独立运行**                   |
| ⑨  | **法律条文搜索**（新增） | `/api/law/*`         | 无独立页面（API 服务） | 法条搜索/验证/法律名称查询，**模块独立运行**                  |

## 1.4 权限模型（三角色）

| 角色               | 权限                                           | 前端界面 |
| ---------------- | -------------------------------------------- | ---- |
| **业务**（business） | 发起咨询、上传材料、查看 AI 意见、提交法务工单、查看自己的工单状态          | 业务端  |
| **法务**（legal）    | 接收工单、审核、**调用知识库检索**（不可直接增删改）、归档前审核修改、校验、确认入库 | 法务端  |
| **管理员**（admin）   | **直接查看/编辑/删除知识库条目**、管理账户、浏览归档、查看全部工单数据       | 管理员端 |

**权限边界说明**：

- 法务**不能直接修改知识库**，可以提交修改申请，也可以在"归档前审核"环节对拟入库条目做修改，经**校验**后确认入库
- 管理员可直接操作知识库（增删改查），拥有最高权限
- 业务只能看到自己提交的工单，法务和管理员可看全部

## 1.5 校验功能（法务归档前）

法务在确认入库前，系统自动对拟入库条目执行 6 项校验：

| 校验项    | 规则                           | 类型 | 失败处理           |
| ------ | ---------------------------- | -- | -------------- |
| 问题完整性  | question 字段非空（**有问题才能有答案**）  | 硬  | 阻止入库，提示补全问题    |
| 四段式完整性 | 四个字段（法律解答/合规风险/实操建议/相关法条）均非空 | 硬  | 阻止入库，提示补全      |
| 法条格式   | `legal_basis` 含《》及"第X条"格式    | 软  | 警告，可强制入库       |
| 标签完整性  | 至少 1 个 tag                   | 软  | 警告，可强制入库       |
| 重复校验   | 与现有知识库条目相似度 > 0.8            | 软  | 警告，显示相似条目供法务判断 |
| 问题格式   | question 以"？"结尾              | 软  | 警告，可强制入库       |

- **硬错误**（errors）：阻止入库
- **软警告**（warnings）：可强制入库（需法务二次确认）

## 1.6 AI 审核引擎需求

| 需求项        | 说明                                                                                     |
| ---------- | -------------------------------------------------------------------------------------- |
| 检索方式       | 双算法：KeywordMatcher 粗筛 Top-20 + TF-IDF 余弦相似度精排 Top-5                                    |
| 生成方式       | RAG + LLM：将检索结果作为上下文，调用大模型生成四段式法务意见                                                    |
| **三级兜底策略** | 高相关度（≥0.45）：知识库 + LLM 优化，标注原文；中相关度（0.15-0.45）：参考知识库 + LLM；低相关度（<0.15）：纯 LLM 兜底，提示未用知识库 |
| **用户反馈机制** | 用户可对引用的知识库条目标注"相关/不相关"，系统统计反馈用于优化检索                                                    |
| 回退机制       | LLM 不可用/失败时，自动回退到规则拼装（纯 RAG）                                                           |
| 大模型        | OpenAI 兼容协议；测试用火山方舟 `ark-code-latest`，生产换私有部署                                          |
| 免责声明       | 每条 AI 回复自动附加免责声明，提示需提交法务获取正式意见                                                         |
| 多轮对话       | 业务可在同一会话内连续咨询，上下文保留                                                                    |
| **材料合并咨询** | 业务可勾选"带上材料提问"，AI 检索结果与上传材料一并作为上下文                                                      |

## 1.7 合同审核模块需求

合同审核模块移植自旧有项目[Contract-Review-System](https://github.com/tidu2020/Contract-Review-System)，作为独立模块嵌入业务端"合同审核"标签页。

| 需求项    | 说明                                                                       |
| ------ | ------------------------------------------------------------------------ |
| 模块独立性  | 自带 `LLMClient`、`AppConfig`，不依赖 `ai`/`knowledge`/`business` 模块；单独注册蓝图即可运行 |
| 数据输入   | 支持上传 `.txt/.md/.docx` 合同文件，自动解析为纯文本                                      |
| 六阶段流水线 | ①合同信息提取 → ②知识检索 → ③业务审查 → ④法务审查 → ⑤法条核验 → ⑥报告生成                          |
| 审核参数   | 特别关注（focus）、审核立场（甲方/乙方/中立/自动识别）                                          |
| 报告输出   | Markdown 格式，含合同概要、风险点、修改建议、法条核验、行动项                                      |
| 多轮问答   | 审核完成后可针对合同细节继续提问，基于已生成报告作答                                               |
| 会话持久化  | 会话以 JSON 文件存储于 `backend/file_review/output/sessions/`，进程重启不丢             |
| LLM 配置 | 通过 `LLM_API_KEY` 环境变量注入；未配置时上传/会话仍可用，但审核会返回 503                          |

## 1.9 法律条文搜索模块需求

法律条文搜索模块移植自旧项目 [law-search](https://github.com/tidu2020/law-search)，作为独立模块提供法条搜索与验证服务。

| 需求项    | 说明                                                                  |
| ------ | ------------------------------------------------------------------- |
| 模块独立性  | 自带 `FLKApiClient` 和 `LawVerifier`，不依赖其他业务模块；单独注册蓝图即可运行              |
| 数据来源   | 国家法律法规数据库 (flk.npc.gov.cn) 真实 API                                   |
| 搜索功能   | 支持关键词搜索、法律名称搜索、自然语言查询                                               |
| 验证功能   | 精确验证特定法律的具体条文（如"民法典第143条"）                                          |
| 别名映射   | 支持常用法律别名（如"民法典"→"中华人民共和国民法典"）                                       |
| 模糊匹配   | 支持自然语言模糊查询（如"偷东西"→"刑法 盗窃罪"）                                         |
| API 路由 | `/api/law/search`（搜索）/ `/api/law/verify`（验证）/ `/api/law/lookup`（查询） |
| 合同审核集成 | `LegalSearcher` 通过 `LawVerifier` 替代原 DuckDuckGo 搜索，增强法律检索准确性        |

## 1.10 非功能需求

- **数据安全**：MVP 阶段知识库本地存储（JSON）；企业部署时合同/函件等涉密材料禁止出公司网络，大模型走私有部署。
- **可演示性**：浏览器访问 `http://localhost:5000`，单条 FAQ 检索响应 < 1 秒（LLM 生成约 3-8 秒）。
- **可维护性**：前后端分离，后端模块化路由，前端按角色独立页面。
- **可扩展性**：知识库存储层抽象，未来可替换为 SQLite/向量库；前端可替换为 React/Vue；大模型可通过环境变量切换。
- **无外部依赖**：后端仅需 Flask，算法用标准库自研实现（符合课程要求）。
- **模块独立性**：各功能模块（auth/business/legal/ai/knowledge/report/archive/file\_review）通过依赖注入解耦，file\_review 模块完全独立运行，不依赖其他业务模块。
