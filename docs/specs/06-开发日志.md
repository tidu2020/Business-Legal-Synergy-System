   - `create_app()` 中初始化 LlmClient 并注入 ReviewOrchestrator
   - 启动时打印 LLM 状态（已启用/未配置）

**遇到的问题与解决**：

| 问题 | 原因 | 解决 |

| 代码改了不生效 | `__pycache__` 缓存旧字节码 | 重启前清理 `__pycache__` |

**验证结果**：
- LLM 连通性测试：✅ 成功，返回专业的第三方付款法律意见
- RAG+LLM 端到端测试：✅ 成功
  - candidates: 20
  - Top1 命中：合同第三方付款需要注意什么？
  - 回复 463 字，四段式结构完整
  - 引用法条：《民法典》第五百一十四条
  - 自动附加免责声明
- 回退机制测试：✅ 4 类异常（401/超时/空响应/未配置）均优雅降级

**关键设计决策**：
- **依赖注入**：LLM 客户端通过构造函数注入 ReviewOrchestrator，便于测试和生产替换
- **优雅降级**：LLM 不可用时不阻断服务，回退纯 RAG 模式，保证系统可用性
- **知识库约束**：System Prompt 明确要求"严格依据知识库案例回答，不得编造"，降低幻觉风险
- **环境变量优先**：生产环境用环境变量注入密钥，不写进源码

### 阶段六：合同审核模块移植 + 多端功能增强 + Bug 修复（2026-06-28）

**目标**：移植 Contract-Review-System 作为独立合同审核模块；按业务/法务/管理员三端需求补齐功能；修复联调 bug；同步更新 specs 文档。

**完成事项**：

#### 1. 合同审核模块移植（file_review，独立模块）

将外部项目 [Contract-Review-System](https://github.com/tidu2020/Contract-Review-System) 移植为 `backend/file_review/` 子系统，**完全独立运行**：

- **目录结构**：`config.py` / `routes.py` / `src/`（pipeline/extractor/case_matcher/legal_searcher/reviewer/report_generator/llm_client/models）/ `prompts/` / `data/` / `output/sessions/` / `tests/`
- **模块自带 LLMClient**（`src/llm_client.py`）：用 `urllib` 标准库重写，不依赖 `openai` SDK，与主系统 `ai/llm_client.py` 完全解耦
  - LRU 缓存（CACHE_MAXSIZE=100）
  - 重试机制（429/超时/5xx，MAX_RETRIES=2）
  - JSON 修复（`_repair_json`：去 markdown 包裹、修尾随逗号、单引号转双引号）
  - 上下文超限检测（ContextLengthExceededError）
  - 输出截断自动扩容（finish_reason=length 时倍增 max_tokens）
- **AppConfig dataclass**（`config.py`）：`LLMConfig.from_central()` 从主系统 config 读取环境变量，但模块内部独立使用
- **独立蓝图**（`routes.py`）：
  - `POST /api/file_review/upload`：上传 .txt/.md/.docx，解析为纯文本并创建会话
  - `POST /api/file_review/run`：执行六阶段流水线
  - `GET /api/file_review/report/<sid>`：下载 Markdown 报告
  - `POST /api/file_review/chat`：基于已审合同的多轮问答
  - `GET /api/file_review/sessions`：会话列表
- **六阶段流水线**（`src/pipeline.py`）：
  1. 合同信息提取（ContractExtractor）
  2. 知识检索（CaseCauseMatcher + LegalSearcher）
  3. 业务审查（review_business）
  4. 法务审查（review_legal）
  5. 法条核验（verify_laws，高风险法条 LLM 复核）
  6. 报告生成（ReportGenerator，Markdown 格式）
- **会话持久化**：JSON 文件存储于 `output/sessions/`，进程重启不丢
- **长合同分块策略**：当 `estimate_tokens > context_limit * 0.75 * 0.85` 时按"第X条"切分，分别审查后合并

#### 2. 业务端功能增强

- **工单撤回**（`business/routes.py`）：法务开始审核前允许业务撤回工单，状态 `submitted → withdrawn`
- **三级 LLM 兜底策略**（`ai/orchestrator.py`）：
  - 高相关度（≥0.45）：知识库 + LLM 优化，标注原文
  - 中相关度（0.15-0.45）：参考知识库 + LLM
  - 低相关度（<0.15）：纯 LLM 兜底，提示未用知识库
- **用户反馈机制**：用户可对引用的知识库条目标注"相关/不相关"，系统统计反馈用于优化检索
- **材料合并咨询**：业务可勾选"带上材料提问"，AI 检索结果与上传材料一并作为上下文
- **前端 business.html**：新增"合同审核"标签页（上传 → 启动审核 → 查看报告 → 多轮问答）

#### 3. 法务端功能增强

- **修改已归档内容**（`legal/routes.py`）：法务可重新编辑自己归档的工单，状态 `archived → reviewing`
- **查看全部知识库**：法务可检索调用全部知识库内容
- **提交知识库修改申请**：法务发现知识库条目有误可提交修改申请，待管理员审批
- **调用 AI 审核工具**：法务在审核工单时可调用 AI 辅助

#### 4. 管理员端功能增强

- **退回归档**（`archive/routes.py`）：管理员可退回归档，状态 `archived → reviewing`，需填写退回原因
- **工单详情查看**：管理员可点开看工单完整内容
- **相似知识合并**（`knowledge/routes.py`）：系统自动检测相似条目并提示合并，合并需法务确认
- **审批知识库修改申请**：管理员通过/驳回法务的修改申请

**关键设计决策**：

| 决策点 | 选择 | 理由 |
|--------|------|------|
| file_review 模块独立性 | 自带 LLMClient + AppConfig，不依赖主系统其他模块 | 满足"所有板块要能够独立工作"要求；可单独移植复用 |
| file_review LLM 实现 | urllib 标准库 | 零外部依赖，符合课程要求；与主系统 LlmClient 解耦 |

| 三级兜底阈值 | HIGH=0.45 / LOW=0.15 | 平衡知识库引用率与 LLM 兜底率，避免低相关度时强行引用知识库 |
| 用户反馈机制 | 记录相关/不相关标注 | 用于优化检索权重，未来可扩展为在线学习 |
| 工单状态机扩展 | submitted/withdrawn/reviewing/confirmed/archived/rejected | 覆盖撤回、退回、重新编辑等流转 |
| 相似合并需法务确认 | 管理员发起合并 → 法务确认 → 真正删除次条目 | 防止误删知识，法务作为内容把关者 |

| 11 | **file_review 模块独立性** | 自带 LLMClient + AppConfig，不依赖主系统其他模块 | 满足"所有板块要能够独立工作"要求；可单独移植复用 |
| 12 | **三级 LLM 兜底** | HIGH=0.45 / LOW=0.15 阈值分流 | 平衡知识库引用率与 LLM 兜底率，避免低相关度时强行引用知识库 |
| 13 | **工单状态机扩展** | submitted/withdrawn/reviewing/confirmed/archived/rejected | 覆盖撤回、退回、重新编辑等流转 |
| 14 | **相似合并需法务确认** | 管理员发起合并 → 法务确认 → 删除次条目 | 防止误删知识，法务作为内容把关者 |
| 15 | **file_review 不回退纯 RAG** | 必须有 LLM 才能运行审核 | 合同审核六阶段流水线强依赖 LLM |
| 16 | **反馈驱动检索排序** | 用户标记"有关/无关" → 加权公式 `得分 × (1.0 + net × 0.15)` | 持续使用中自动优化检索准确率，无需人工干预 |

**完成事项**：

1. **后端：法务端合并确认 API**
   - `GET /api/legal/merge_requests`：列出合并申请（支持 status 筛选）
   - `GET /api/legal/merge_requests/<req_id>`：申请详情（含主/次条目快照 + 合并建议稿）
   - `POST /api/legal/merge_requests/<req_id>/confirm`：法务确认 → 应用 merged_patch 至主条目 + 删除次条目 + 刷新检索索引
   - `POST /api/legal/merge_requests/<req_id>/cancel`：取消申请
   - 修复 `merge_items` 端点：不再立即应用补丁，改为存入申请记录待法务确认

2. **后端：归档自动总结入库申请**
   - 重写 `archive/orders/<order_id>/archive` 端点：归档前调用 LLM 自动总结问答生成 FAQ
   - 新增 `_generate_kb_submit_request` 辅助函数：
     - 调用 `orchestrator.llm_json` 生成结构化 FAQ
     - LLM 不可用时回退到业务问题 + 法务结论拼接
     - 检索现有知识库做冲突检测（相似度≥0.75 标记 conflict）
     - 写入 `data/kb_submit_requests.json`，状态 pending

3. **后端：管理员审批入库申请 API**
   - `GET /api/kb/kb_submit_requests?status=`：列出申请
   - `POST /api/kb/kb_submit_requests/<req_id>/approve`：批准 → FAQ 入库 + 刷新索引
   - `POST /api/kb/kb_submit_requests/<req_id>/reject`：驳回（带原因）

4. **后端：相似对自动扫描 + LLM 合并建议稿**
   - `GET /api/kb/similar_pairs`：扫描全库相似对（相似度≥0.60）
   - `POST /api/kb/merge_draft`：调用 LLM 生成合并建议稿（含 question/legal_answer/compliance_risk/practical_advice/legal_basis/category/tags）
   - LLM 不可用时回退字段拼接生成建议稿

5. **前端 admin.html：相似合并 tab 重写**
   - 自动扫描按钮：列出全库相似对（A/B 对比卡片 + 相似度）
   - 生成建议稿：弹窗显示 LLM 生成的合并稿，所有字段可编辑
   - 提交合并申请：携带管理员修改后的 merged_patch，待法务确认

6. **前端 admin.html：入库申请审批 UI**
   - 新增"入库申请"侧栏项
   - 列表展示：申请号/状态/提交人/关联工单/FAQ 详情/冲突提示
   - 批准/驳回按钮 + 状态历史展示

   - 列表展示：主/次条目对比 + 合并建议稿预览
